# CLAUDE.md — project init & handoff notes

Read this first when picking up this project on a new machine.

## What this is

A reproducible harness evaluating **Turso** (Rust SQLite rewrite, pinned `v0.6.1`) as a
distributed **edge OLTP** database feeding an **OLAP/lakehouse** layer (DuckDB + Apache Iceberg)
via **Change Data Capture**, plus native **master→replica** replication and an **edge vector
search** demo. See `README.md` for the overview and `docs/REPORT.md` for the written results.

**This harness is implemented in Rust** (single `cascade` binary). It was originally a Python harness
driving the `tursodb` CLI + `pyturso`; it was migrated to native Rust — see *Rust migration* below.

## Init on a fresh machine (do this first)

```bash
./setup.sh   # detects OS/arch, downloads the tursodb CLI into .work/bin, then `cargo build --release`
./run.sh     # runs the whole pipeline on SYNTHETIC data, writes docs/REPORT.md
```

That's it — **no external data needed**; synthetic data is generated by default. Everything is
config-driven (see env table in `README.md`); there are **no hardcoded machine paths**.

### Requirements
- macOS (arm64/x86_64) or Linux/WSL2. **Rust ≥ 1.94** (`cargo`), `bash`, `curl`, `tar` (+xz), `git`.
- **Windows native does NOT work** — the prebuilt `tursodb` sync-server CLI targets macOS/Linux
  and Turso's `io_uring` is Linux-only. On a Mac it runs natively.
- DuckDB is built **bundled** by the `duckdb` crate (no system DuckDB). First build ~3–5 min.

## Rust migration (what we did)

Migrated the Python harness to a native-Rust single binary (`src/`, package `cascade`,
bin `cascade`). Why it's possible/better here: Turso is itself a Rust project, so on macOS/Linux we
use the crates directly instead of the `pyturso` binding + a Python venv.

Crate choices and the gotchas that cost iterations (so you don't re-discover them):
- **`turso = "0.6.1"` (features `["sync"]`)** — max-stable crate, **exact** protocol match to the
  `tursodb` v0.6.1 CLI used as the `--sync-server`. Async/tokio. Local: `Builder::new_local(path)`
  → `db.connect()` (sync). Sync: `turso::sync::Builder::new_remote(path).with_remote_url(url)
  .bootstrap_if_empty(true)`; `db.connect()` here is **async**; `db.push()/.pull()->bool/.stats()`.
  `stats()` fields: `network_sent_bytes`, `network_received_bytes`, `main_wal_size`. CDC + vector
  are SQL-level (`PRAGMA capture_data_changes_conn`, `vector32`/`vector_distance_cos`) — they work
  through `execute`/`query`, no special crate API.
- **`duckdb` (bundled)** — kept **arrow-free** (Appender/SQL + `params_from_iter`) so DuckDB's
  internal arrow (58) never has to interop with the rest of the arrow graph.
- **`iceberg = "0.9.1"` + `iceberg-catalog-sql = "0.9.1"`** — real Iceberg tables via a sqlite
  `SqlCatalog`. **This forces `arrow`/`arrow-array`/`arrow-schema`/`parquet` = "57"** (iceberg's
  arrow major); pinning 55 silently creates a second incompatible arrow and the writer rejects
  RecordBatches. Also: `sqlx` must force-enable `runtime-tokio`; `SqlCatalogBuilder` needs
  `.with_storage_factory(Arc::new(LocalFsStorageFactory))`; create tables from arrow via
  `arrow_schema_to_schema_auto_assign_ids` (NOT the non-auto variant); build the write RecordBatch
  from `schema_to_arrow_schema(table.metadata().current_schema())` (embeds `PARQUET:field_id`);
  commit via `Transaction::fast_append().add_data_files(..).apply(tx)?; tx.commit(&catalog)`.
- **Toolchain:** iceberg 0.9.1 needs rustc ≥ 1.92, but its `fastnum` dep needs **1.94** — so the
  effective floor is **1.94**. `rustup update stable`.

Behavior deltas vs the Python harness (intentional / unavoidable):
1. **Synthetic data isn't byte-identical** — Rust's RNG ≠ Python's Mersenne Twister even with the
   same seed. Distributions match; the real-data path (`PATENTS_JSONL`) is the reproducible one.
2. **Storage measurement now checkpoints first.** `cdc-overhead` runs `PRAGMA wal_checkpoint
   (TRUNCATE)` before sizing the db, then sums main + sidecars (`db_footprint`). The Python harness
   measured only the post-close main file, which was noisy/understated. Amplification reads ~2×.
3. **Sync metadata cleanup.** The sync engine writes `<db>-info`, `<db>-changes`,
   `<db>-wal-revert` next to the db. `remove_db_files` deletes these too — stale metadata without a
   main db makes the sync engine fail "main DB file doesn't exist, but metadata is".

## KNOWN GAPS / MISSING THINGS (important)

1. **Real dataset is NOT in the repo.** The headline numbers came from a 1.5 GB patents JSONL that
   only exists on the original dev box. It is USPTO energy-storage patents. To reproduce real
   numbers, supply **any** JSONL with these fields and point `PATENTS_JSONL` at it:
   `patent_id, title, abstract, claims, cpc_codes[], assignee, grant_date, cited_patent_ids[]`.
   Otherwise the synthetic generator (`cascade gen-synthetic`) stands in.
2. **`results/*.json` are overwritten on every run.** Running `./run.sh` with synthetic data
   replaces them with synthetic numbers. Copy `results/` aside first to keep a baseline, or only
   run with `PATENTS_JSONL` set.
3. **Version lockstep.** The `turso` crate (`Cargo.toml`) and the CLI (`TURSO_VERSION` in
   `setup.sh`) must match — the sync wire protocol is version-sensitive. Both are `0.6.1`.
4. **CDC ⊕ MVCC are mutually exclusive.** You cannot demo `BEGIN CONCURRENT` write throughput
   and CDC in the same connection/run. They are two separate experiments.
5. **Vector search is linear-scan, no ANN index** in v0.6.1. Latency grows ~linearly with rows.
   Honest only for small/medium edge datasets.
6. **Sync is one-way primary→replica**, no conflict resolution / multi-primary. "Master-slave"
   is accurate; "multi-master" is not. Edge writes have a *constrained* out-of-band path:
   `cascade enqueue`/`flush` ship a local `<db>.outbox` to the master gateway `/inbox` (queue-based,
   last-writer-wins on `inbox` PK), which then replicates back out. Not conflict-free.
7. **Turso is BETA** (project's own warning) — frame reliability claims accordingly.
8. **macOS `tar` + `.xz`:** bsdtar usually handles `.tar.xz`. If `setup.sh` fails extracting,
   `brew install xz` and retry.
9. **Comparison to current standards** — `cascade compare-cdc` now measures built-in CDC vs the
   hand-rolled SQLite trigger pattern (same engine, runnable today). The full cross-engine pipeline
   (SQLite+Litestream, Postgres+Debezium) is scaffolded in `docker/compare/` — validate on Linux
   and quote the numbers; that's still the most-wanted contribution.

## Turso API reference (SQL-level; unchanged by the language port)

**CDC** (per-connection):
```sql
PRAGMA capture_data_changes_conn('full');   -- modes: off | id | before | after | full
-- changes land in table turso_cdc:
--   change_id, change_time, change_type (1=INSERT,0=UPDATE,-1=DELETE,2=COMMIT),
--   table_name, id, before(BLOB), after(BLOB), updates(BLOB), change_txn_id
-- decode the binary row images to JSON via SQL helpers:
SELECT bin_record_json_object(table_columns_json_array(table_name), after) FROM turso_cdc;
-- cursor/resume: WHERE change_id > ? ORDER BY change_id LIMIT N
```

**Native replication / sync:**
```bash
tursodb primary.db --sync-server 0.0.0.0:8080   # the "master" hub (prebuilt CLI has this flag)
```
```rust
use turso::sync::Builder;
let db = Builder::new_remote("replica.db").with_remote_url("http://host:8080")
    .bootstrap_if_empty(true).build().await?;
let conn = db.connect().await?;       // NOTE: async on synced dbs
conn.execute(/* ... */, ()).await?;
db.push().await?;                      // send local writes up
db.pull().await?;                      // pull remote changes down (-> bool)
let s = db.stats().await?;             // s.network_sent_bytes / network_received_bytes / main_wal_size
```

**Vector / edge AI:**
```sql
CREATE TABLE docs(id TEXT, emb F32_BLOB(64));
INSERT INTO docs VALUES ('x', vector32('[0.1,0.2,...]'));
SELECT id, vector_distance_cos(emb, vector32('[...]')) d FROM docs ORDER BY d LIMIT 10;
-- also: vector64/vector8/vector1bit, vector_distance_l2/dot/jaccard
```

## File map

```
Cargo.toml               crate deps (turso 0.6.1 [sync], duckdb bundled, iceberg 0.9.1, arrow 57)
setup.sh                 portable bootstrap (CLI download + cargo build --release)
run.sh                   build-if-needed + `cascade run-all`
src/
  main.rs                clap CLI (one subcommand per phase) + run-all orchestration + sync server spawn
  common.rs              paths/env, arrow schemas, Parquet IO, turso Value conversion, save_result, db_footprint, run_id
  sink.rs                pluggable CDC sinks (trait Sink + DuckDb/Webhook/Jsonl), used by node.drain_olap
  compare.rs             competitor bench: built-in CDC vs trigger CDC vs none (`cascade compare-cdc`)
  gen_synthetic.rs       Phase 1: synthetic patents/cpc/citations Parquet (no external dataset)
  prep_data.rs           Phase 1: parse a real patents JSONL -> Parquet (PATENTS_JSONL)
  cdc_overhead.rs        Phase 3a
  cdc_to_olap.rs         Phase 3b (DuckDB + real Iceberg table)
  replication.rs         Phase 2 (turso::sync push/pull/stats)
  olap.rs                Phase 4
  vector.rs              Phase 5
  report.rs              Phase 6 -> docs/REPORT.md
results/                 committed result JSONs (overwritten each run)
docs/REPORT.md           generated writeup
.work/  target/          runtime artifacts / build output — gitignored
```

## Running individual phases

```bash
./target/release/cascade --help            # list subcommands
./target/release/cascade gen-synthetic 20000
./target/release/cascade cdc-overhead
# replication needs a running sync server; run-all spawns it. Standalone:
.work/bin/*/tursodb .work/db/repl_server.db --sync-server 127.0.0.1:8080 &
./target/release/cascade replication
```
